# Connector `tasks.max` 설정 가이드

## 핵심 요약

```
============================================================
 [ 1순위 고려사항 ]
 1) Connect 클러스터 내 부하 분산이 되도록 task 수를 적절히 분배
 2) task 수가 코어 수를 초과하지 않도록 하여
    컨텍스트 스위치 비용 최소화
============================================================
```

---

## 공통 원칙

### task 수에 따른 트레이드오프

| task 수 | 장점 | 단점 |
|---|---|---|
| 많음 | Connect 클러스터 부하분산 효과 ↑ | 코어 수 초과 시 컨텍스트 스위치 비용 발생 |
| 적음 | 컨텍스트 스위치 비용 ↓ | 부하분산 효과 ↓ |

### Connect 클러스터 내 커넥터 수에 따른 전략

- 커넥터 수가 적을 때
  - `tasks.max`를 Connect 노드 수만큼 지정
  - → Connect의 round-robin 분배로 노드 간 균등한 부하분산 기대
- 커넥터 수가 많을 때
  - 이미 커넥터 단위로 부하가 분산되어 있음
  - → 커넥터 당 task 수를 낮춰서 컨텍스트 스위치 비용 완화

---

## 커넥터 유형별 가이드

### JDBC Source 등 DB 연동 계열

- 테이블 1개에는 task 1개만 붙을 수 있음 (1개 테이블을 여러 task로 쪼개서 병렬처리 불가)
- 병렬화 상한: `tasks.max` ≤ 대상 테이블 수
  - task 수가 테이블 수보다 적으면 → 1개 task가 여러 테이블을 담당 (순차 처리)
  - task 수가 테이블 수보다 많으면 → 초과분은 유휴 상태
- 테이블 수에 맞춰 task 수를 늘리면 처리 속도 ↑ (상한 내에서)

### S3 Sink 등 Sink 계열

- 병렬화 상한: `tasks.max` ≤ 대상 토픽들의 총 파티션 수 (Consumer Group 기반)
  - task 수가 파티션 수보다 적으면 → 1개 task가 여러 파티션을 담당
  - task 수가 파티션 수보다 많으면 → 초과분은 유휴 상태 (할당받을 파티션 없음)
- 파티션,task 수에 비례하여 처리 병렬도 증가 (상한 내에서)
  - 단, 실제 전송 속도는 네트워크·Sink 대상(S3 API 등)의 처리량에 의해 제한될 수 있음
