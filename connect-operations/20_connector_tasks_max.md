# Connector `tasks.max` 설정 가이드

## 핵심 요약

> ## 🚨 **\[ 1순위 고려사항 \]** 🚨
>
> ### 1) **Connect 클러스터 내 부하 분산**이 되도록 task 수를 적절히 분배
> ### 2) task 수가 **코어 수를 과도하게 초과하지 않도록** 하여 **컨텍스트 스위치 비용 최소화**

---

## 공통 원칙

### task 수에 따른 트레이드오프

| task 수 | 장점 | 단점 |
|---|---|---|
| 많음 | Connect 클러스터 부하분산 효과 ↑ | 코어 수를 많이 초과하면 컨텍스트 스위치 비용 발생 |
| 적음 | 컨텍스트 스위치 비용 ↓ | 부하분산 효과 ↓ |

### Connect 클러스터 내 커넥터 수에 따른 전략

- 커넥터 수가 적을 때
  - `tasks.max`를 Connect 노드 수만큼 지정
  - → Connect의 round-robin 분배로 노드 간 균등한 부하분산 기대
- 커넥터 수가 많을 때
  - 이미 커넥터 단위로 부하가 분산되어 있음
  - → 커넥터 당 task 수를 낮춰서 컨텍스트 스위치 비용 완화

---

## 커넥터 유형별 가이드

### JDBC Source 등 DB 연동 계열

- 테이블 1개에는 task 1개만 붙을 수 있음 (1개 테이블을 여러 task로 쪼개서 병렬처리 불가)
- 병렬화 상한: `tasks.max` ≤ 대상 테이블 수
  - task 수가 테이블 수보다 적으면 → 1개 task가 여러 테이블을 담당 (순차 처리)
  - task 수가 테이블 수보다 많으면 → 초과분은 유휴 상태
- 테이블 수에 맞춰 task 수를 늘리면 처리 속도 ↑ (상한 내에서)

### S3 Sink 등 Sink 계열

- 병렬화 상한: `tasks.max` ≤ 대상 토픽들의 총 파티션 수 (Consumer Group 기반)
  - task 수가 파티션 수보다 적으면 → 1개 task가 여러 파티션을 담당
  - task 수가 파티션 수보다 많으면 → 초과분은 유휴 상태 (할당받을 파티션 없음)
- 파티션,task 수에 비례하여 처리 병렬도 증가 (상한 내에서)
  - 단, 실제 전송 속도는 네트워크·Sink 대상(S3 API 등)의 처리량에 의해 제한될 수 있음

## 물리장비 관점의 tasks.max 튜닝

### htop 지표 참고
`tasks.max`를 조정했을 때 `htop`의 `CPU 사용률`·`Load Average`·`thr` 같은 지표는 부하 변화를 가늠하는 **비례 지표**로 활용할 수 있다.

### EC2 세대
`동일 체급(동일 vCPU·메모리)이라도 EC2 세대를 올리면` 동일 부하를 더 여유있게 처리한다. 특히 `커넥터 task 수가 많을 때 컨텍스트 스위치 이슈에서 꽤나 체감되는 유의미한 효과`를 보았다.
- 사례: 커넥터task 1500개인 커넥트 1대에서, c6i=>c8i 변경시 상시 cpu 점유가 절반으로 감소했으며, 메모리 점유가 튀는 현상이 사라짐. 특히 데이터 처리량이 적을때도 6세대에선 메모리 반환이 더딘 경우가 많았는데, 8세대에선 즉시 반환되어 점유율이 감소. 
- 7세대(m7i,c7i,r7i): 직전 세대 대비 CPU성능 약 15% 개선, DDR5 메모리 도입
- 8세대(m8i,c8i,r8i): 직전 세대 대비 CPU성능 약 20% 개선

동시 실행 task가 많은 Connect 워커일수록 세대 업그레이드의 체감 이득이 크다.

#### 참고: (용어 주의) "task"라는 단어가 층위마다 다르게 쓰임

| 용어 | 층위 | 정의 | 비고 |
|---|---|---|---|
| **Kafka Connect `task`** | 애플리케이션 | 커넥터 내부의 논리적 작업 단위. 하나의 Connect Worker JVM 안에서 스레드 묶음으로 실행됨 | `tasks.max`가 제어하는 대상 |
| **Linux 커널의 `task`** | OS 커널 (Linux 구현) | 스케줄 단위 (`task_struct`). Linux는 프로세스/스레드를 구분하지 않고 모두 `task_struct`로 다룸 — CS 이론의 process/thread 구분과는 다른 Linux 특유의 명명 | 실질적으로 **스레드**에 대응 (= htop의 `thr`) |
| **htop의 `Tasks`** (상단 좌측) | htop UI | 시스템 전체 **프로세스** 수 (TGID 기준) | Linux 커널 용어의 task와는 다름 — UI 레이블일 뿐 |
| **htop의 `thr`** (Tasks 옆) | htop UI | 시스템 전체 **스레드** 수 | Linux 커널 용어의 task에 대응 |

핵심: 단어가 같다고 1:1 대응되지 않는다.
* Kafka Connect `task` 1개는 JVM 스레드 여러 개로 실행되므로, `tasks.max`를 1 늘려도 htop의 `thr`은 커넥터 구현에 따라 **수 개 단위**로 증가한다.
* 따라서 htop 지표는 `tasks.max` 튜닝의 **절대 공식이 아니라 경향 지표**로만 참고한다.
